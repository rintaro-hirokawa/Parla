# 実装方針

作成日: 2026-04-10

---

## 1. 全体アプローチ

### コードが設計になる

詳細設計書を別途作成しない。要件定義（docs/requirements/）とアーキテクチャ方針（docs/architecture/overview.md）を前提に、コードとテストが詳細設計の役割を担う。

- **要件（What）**: docs/requirements/ に定義済み。安定している
- **アーキテクチャ方針（Howの大枠）**: docs/architecture/overview.md に定義済み。安定している
- **詳細設計（Howの詳細）**: コードとテストで表現する。中間成果物としての設計書は作らない

### 垂直スライス

全エンティティ → 全Port → 全Adapter と水平に進めるのではなく、1つのユーザーフローを端から端まで通すことを1単位として進める。

### UIは後

各スライスの動作確認はテストとCLIで行う。UIは全スライスが揃ってから被せる。ドメインが安定していればUIは薄いレイヤーなので、後から作っても手戻りが少ない。

## 2. テスト方針

### ドメインロジックにはTDD

SRS計算、セッション構成ルール、ヒントレベル判定、評価基準など、入力と出力が要件定義に定義されているものはテストを先に書く。テストが仕様書になり、実行すれば正しさが検証される。

### TDDを適用しない領域

| 領域 | 理由 | 代わりに |
|---|---|---|
| Adapter（LLM呼び出し等） | 出力が非決定的 | スナップショットテスト（実レスポンスを保存して検証） |
| UI | pytest-qt で TDD | ViewModel、ウィジェット、ナビゲーション、View をテスト |
| 探索的なコード | インターフェースが未定 | 後からテスト追加 |

### 結合テスト

スライスごとに、パーツ間の接続を検証する結合テストを書く。パーツ単体のユニットテストだけでは、型の不一致や変換漏れ等の接続ミスを検知できない。

**方針:**
- 外部APIは呼ばない。LLM・TTS・Azure等はFake実装に差し替える
- SQLiteは本物を使う（`:memory:` で十分）
- EventBusの配線も本番と同じにする
- Fakeのレスポンスデータは verification/ の実行結果JSONからfixtureを構成する
- 外部APIのFake化が難しいケースは都度判断する

**例（スライス1）:**
```
register_source → emit(SourceRegistered) → handler → Fake LLM → SQLite保存 → ステータス確認
```

## 3. 実装順序

### 事前準備（スライス横断の基盤）

スライスを切り始める前に、全スライスが使う基盤を決める。

- **ディレクトリ構成の方針確定**: スライス1で作るファイルの置き場所が必要
- **Event Busの実装**: 全スライスが使う共通基盤
- **エラーハンドリング方針**: セッション中の障害時にどうするか（1ページ程度の方針）

### スライス一覧

各スライスの中身（エンティティ、Port、イベント等）は実装時にコードとして設計する。

| 順 | スライス | 対象要件 | 選定理由 |
|---|---|---|---|
| 1 | ソース登録 → パッセージ生成 | 04, 10(#1), 07 | 最もシンプルな完結フロー。アーキテクチャの骨格が立つ |
| 2 | Phase A → Phase B（フィードバック） | 02, 10(#4,#5), 05, 12 | アプリの心臓部。ここが動けば価値が見える |
| 3 | Block 1 レビュー（SRS） | 06, 03, 10(#3), 07 | SRSが入ると「学習アプリ」になる |
| 4 | Phase C（通し練習） | 02, 12, 11 | 外部サービス統合が最も多いスライス |
| 5 | セッション構成 + メニュー確定 | 03, 11 | 全体を1日のセッションとして統合 |

### 各スライスの進め方

1. 対象フローの要件を再読する（docs/requirements/ から）
2. 関連する技術検証（verification/）を特定し、検証結果と教訓を確認する。特にLLMの精度に関わる知見（プロンプト、モデル名・パラメータ、バリデーション戦略、LLMの癖への対処法）は Adapter 実装に最大限活用する。ただしデータ構造はドメインモデルと Port の設計に従い、検証のスキーマをそのまま流用しない
3. ドメインモデルをコードとして書く（TDD対象はテスト先行）
4. Port インターフェースを定義する（このフローで必要なものだけ）
5. Adapter を実装する
6. イベントを定義してハンドラを繋ぐ
7. DB スキーマを作る（このフローで必要なテーブルだけ）

### 動作確認の粒度

- 基本はテスト（pytest）で検証する。ドメインロジックのTDDテストが主な確認手段
- 必要に応じて簡易スクリプト（`if __name__ == "__main__"` レベル）で手動確認する。verification/ と同じ粒度感
- CLIフレームワーク（Typer等）は導入しない。成果物としてのCLIは作らない

### UI統合（全スライス完了後）

画面一覧と遷移フローは [09-ux-screens.md](../requirements/09-ux-screens.md) に定義済み。ワイヤーフレームは作らず、PySide6で直接実装して触りながら調整する。
