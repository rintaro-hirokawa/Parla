## 1. はじめに

### 目的

本ドキュメントは AI-Driven-English アプリケーションの実装アーキテクチャを定義する。要件定義（[docs/requirements/](../requirements/README.md)）で定めた機能を、保守性・拡張性・テスト容易性を保ちながら実現するための構造的な指針を示す。

### 前提

- **プロトタイプフェーズ**: 機能の追加・削除・改善が頻繁に行われる。過度な抽象化は避け、実装速度を優先する
- **将来のモバイル移行**: デスクトップ（PySide6）→ API サーバー（FastAPI）+ モバイルクライアント（SwiftUI）への移行を見据える。ドメインロジックとAIオーケストレーションはサーバー側でそのまま再利用する
- **単一開発者 + コーディングエージェント**: 人間とAIエージェントの双方がコードを理解・修正できる構造にする

### このアプリの構造的な難しさ

このアプリケーションには3つの軸が交差する本質的な複雑さがある:

| 軸 | 内容 | 例 |
|---|---|---|
| ドメインロジック | ルールベースで決定論的な処理 | SRS間隔計算、セッション構成、定着判定、ヒントレベル制御 |
| AIオーケストレーション | LLMコールの管理・プロンプト制御 | 6種のLLMコール（[10-llm-calls.md](../requirements/10-llm-calls.md)）、事前生成/リアルタイム/イベント駆動の混在 |
| 外部サービス依存 | 複数の外部APIの統合 | Gemini（LLM）、ElevenLabs（TTS）、Azure Speech Service（発音評価）、音声デバイス |

これらが密結合すると「SRSロジックを直したいだけなのにLLMコールのコードを触る必要がある」状態に陥る。本アーキテクチャの第一目的は、これら3軸を構造的に分離することである。

## 2. 設計原則

### 2.1 依存性逆転

ドメイン（ビジネスロジック）が最も重要であり、最も安定している。ドメインは外部の世界（UI、LLM、DB、音声デバイス）に一切依存しない。外部がドメインに依存する。

```
  ✗ Domain → Gemini API     （ドメインが外部を知っている）
  ✓ Domain → Port ← Adapter → Gemini API  （ドメインはPortだけ知っている）
```

### 2.2 段階的表示（Progressive Rendering）

処理結果は完了次第UIに反映する。全結果の完了を待たない。完了した部分結果をドメインイベントとしてEvent Busに発火し、ViewModelが逐次UIに反映する。これはセクション5のイベント駆動設計の自然な拡張であり、新しい仕組みの導入ではない。

### 2.4 プロトタイプの実用主義

構造的に後から直すのが困難なものだけを今守り、それ以外はシンプルに保つ。

**今やること（後から直すのが困難）:**

- ドメインコアを外部依存から分離するディレクトリ構造
- LLMコールをPortの裏に隠す（ドメインにべた書きしない）
- ドメインイベントの仕組み（シンプルなインメモリ pub/sub で十分）
- UIとドメインの分離（UIフレームワークのコードにビジネスロジックを書かない）

**今やらないこと（プロトタイプが安定してから）:**

- リポジトリパターンによるDB抽象化の厳密な実装
- 厳密な集約境界の設計（まず動くモデルを作り、境界は後から調整）
- CQRS（コマンド/クエリ分離）の厳密な実装
- 非同期メッセージキュー（インメモリの非同期処理で十分）

ただし、UI の一覧・履歴・集計・起動時判定のための**読み取り専用 Query Service**は許容する。これは strict CQRS の導入ではなく、既存の application / service 層の中で read-only な責務を分離するだけである。

## 3. 全体構造：ヘキサゴナルアーキテクチャ

### 3.1 概要

ヘキサゴナルアーキテクチャ（Ports & Adapters パターン）を採用する。ドメインを中心に配置し、外部世界との接点を Port（インターフェース）として定義する。Port の具体的な実装が Adapter である。

```
┌──────────────────────────────────────────────────┐
│                                                    │
│  外部サービス: Gemini / ElevenLabs / Azure Speech  │
│  クライアント: UI + 音声デバイス                      │
│                                                    │
│         ▼ Adapter層（外部サービスごと/用途ごと）       │
│         ▼ Port層（ドメインが必要とする契約）           │
│  ═══════════════════════════════════════════════    │
│         ▼                                          │
│  ┌──────────────────────────────────────────┐      │
│  │           Domain Core（内側）               │      │
│  │                                          │      │
│  │  ※ドメインは外部サービスを知らない          │      │
│  │    Port を通じて「欲しい結果」を要求するだけ │      │
│  └──────────────────────────────────────────┘      │
└──────────────────────────────────────────────────┘
```

この図はヘキサゴナルの層構造のみを示す。Port の具体的な粒度や Adapter の分割方針は詳細設計で決定する（6.1節参照）。

**音声デバイスとサーバー側サービスの違い:**
Gemini・ElevenLabs・Azure Speech Service はサーバー側のサービスであり、Portで抽象化してAdapter差し替えが可能（Provider変更に対応）。一方、音声デバイス（マイク・スピーカー）はクライアント側のハードウェアであり、プラットフォームごとに全面書き直しになる。そのため、Portとして抽象化する意味がなく、クライアント層（UI）に含める。ドメインは録音結果を `AudioData`（バイト列 + フォーマット + 長さ）という値オブジェクトとして受け取るだけで、録音手段を知らない。

### 3.2 採用理由

| 恩恵 | このプロジェクトでの意味 |
|---|---|
| UI差し替え | PySide6 → FastAPI への移行時、ドメインとAIオーケストレーションをそのまま持っていける |
| LLMプロバイダ差し替え | Gemini → Claude 等の変更がAdapter層で完結する |
| テスト容易性 | ドメインロジックをLLMやDBなしで単体テストできる |
| プロンプト改善の独立性 | LLMのプロンプト変更がドメインロジックのテストを壊さない |

### 3.3 層の構成と依存方向

```
依存の方向: 外側 → 内側（内側は外側を知らない）

UI ──→ Domain   ✓（UIがドメインを呼ぶ）
Domain ──→ UI   ✗（ドメインはUIを知らない）
Adapter ──→ Port ✓（AdapterがPortを実装する）
Domain ──→ Port  ✓（ドメインはPortを通じて外部と話す）
Domain ──→ Adapter ✗（ドメインは具体的な実装を知らない）
```

### 3.4 ディレクトリ構成

詳細設計で決定する。ドメインモデルの設計結果から導出する方針であり、事前にディレクトリ構成を固定しない。

確定している方針:
- ドメイン層、Port/Adapter層、UI層は別ディレクトリに分離する
- アプリケーション層のユースケースは `services/` に置く。状態変更を行う command service と、読み取り専用の query service は同じ service 層に置いてよい
- ドメインは単一モジュールとし、ファイル単位で論理的に分類する（4章参照）

### 3.5 UI統合: PySide6.QtAsyncio とプログレッシブ表示

#### 統合イベントループ

`PySide6.QtAsyncio` を使用し、Qt のイベントループが Python の asyncio イベントループを兼ねる。アプリケーション全体で**単一のイベントループ**が動作する。

```python
import PySide6.QtAsyncio as QtAsyncio

# Qt のイベントループ = asyncio のイベントループ
QtAsyncio.run(main())
```

これにより:
- 非同期イベントハンドラの asyncio タスク（LLMコール、TTS生成等）が Qt ループ上で実行される
- **別スレッド不要**。async/await と Qt Signal が同一スレッド上で自然に共存する
- イベントハンドラ内で Qt Signal を emit しても、スレッド間の同期問題が発生しない

#### ViewModel とプログレッシブ表示

各画面に対応する ViewModel がドメインと UI の橋渡しを行う。ViewModel はEvent Bus のハンドラとして登録され、ドメインイベントを受け取るたびに内部状態を更新し、Qt Signal で UI に通知する。

```
非同期イベントハンドラ: sentence_1 のフィードバック生成完了
    │
    ├── event_bus.emit(FeedbackReady(sentence_id=1, feedback=...))
    │
    ↓ Event Bus（セクション5参照）
    │
    ├── [ViewModel ハンドラ] 状態更新 → Qt Signal emit → UI描画
    ├── [ストックハンドラ] 学習項目ストック処理
    └── [Logging ハンドラ] ログ記録
```

- 非同期ハンドラは結果を Event Bus に放送するだけ。誰が受け取るか知らない
- ViewModel はイベントハンドラとして部分結果を受け取り、Qt Signal で画面を逐次更新する
- 既存の Event Bus 設計（セクション5）をそのまま活用する。新しい仕組みの導入は不要

#### 具体例: フェーズA→B フィードバックフロー

1. ユーザーがセンテンス1〜8を発話（Phase A）
2. 各センテンス発話完了で `SentenceRecorded` イベント発火
3. 非同期イベントハンドラが各センテンスのフィードバック生成を asyncio タスクとして開始
4. 生成完了のたびに `FeedbackReady` イベントを emit
5. ViewModel ハンドラが受け取り → Qt Signal → UI に即座に表示
6. ユーザーはセンテンス8の完了を待たずにセンテンス1のフィードバックを見られる

## 4. ドメインモデルの関心事

### 4.1 設計方針：単一ドメインモジュール

ドメインは**単一モジュール**として構成する。境界づけられたコンテキスト（Bounded Context）による分割は行わない。

#### 判断の経緯

当初は4コンテキスト、次に2コンテキスト（Knowledge / Session）を検討したが、以下の理由から単一ドメインに統合した:

- **中核フロー（Phase B フィードバック）が全エンティティを横断する**: 入力（Passage + 音声 + 既存学習項目）と出力（動的模範解答 + 新規学習項目 + 練習記録）が全領域にまたがり、BC境界を設けると毎回跨ぐコストが発生する
- **プロトタイプ期はモデルが頻繁に変わる**: エンティティの属性や関係が安定する前にBC境界を引くと、境界の引き直しが設計作業の主要コストになる
- **単一開発者+エージェント**: BC間の契約管理のオーバーヘッドに見合うチーム規模ではない

将来、コードベースが成長しモデルが安定した段階で、実コードの凝集度に基づいてBC分割を検討する。

#### 守るべき境界

BC境界は設けないが、以下の境界は厳守する:

- **ヘキサゴナルの外周**: ドメイン ↔ インフラ（Port/Adapter）の分離。これが最重要
- **ファイル単位の論理的分類**: 関心事ごとにファイルを分ける（素材系、学習項目系、セッション系）

### 4.2 ドメインの関心事

ドメイン内の主要な関心事を3つに分類する。これはファイル分割の指針であり、モジュール境界ではない。エンティティ、サービスの具体的な定義は詳細設計で行う。

- **素材**: 「何を学ぶか」の管理（ソース、パッセージ、センテンス等）
- **学習項目・SRS**: 「どれだけ身についているか」の管理（学習項目、SRS状態、練習履歴等）
- **セッション**: 「今日の学習を駆動する」実行エンジン（セッション、ブロック、フェーズ進行等）

これらに加え、進捗集計（セッションサマリー、WPMトレンド等）は独自のビジネスルールを持たない読み取り専用のクエリ群として扱う。実装上はドメインではなく application / service 層に置き、Repository Port を通じて集約した UI 向け DTO を返す。

## 5. イベント駆動アーキテクチャ

### 5.1 ドメインイベントの役割

このアプリはバックグラウンドでの先読み生成（[11-latency-strategy.md](../requirements/11-latency-strategy.md)）が生命線である。「何をトリガーに何を事前生成するか」はプロトタイプ中に頻繁に変わるため、トリガーと後続処理を疎結合にする必要がある。

イベント駆動にしない場合、フィードバック完了処理はこうなる:

```python
# 悪い例: フィードバックが後続処理を全て知っている
def complete_feedback(self, passage_id):
    feedback = self.generate_feedback(passage_id)
    self.tts_service.generate(feedback.dynamic_answer)
    self.coherence_service.regenerate(feedback)
    self.variation_service.generate(feedback.new_items)
    self.progress_service.update(feedback)
```

イベント駆動にすると:

```python
# フィードバックは「終わった」と言うだけ
def complete_feedback(self, passage_id):
    feedback = self.generate_feedback(passage_id)
    self.emit(FeedbackCompleted(passage_id, feedback))
```

後続処理はハンドラとして独立に登録・削除できる。フィードバックのコードに触れずにバックグラウンド生成戦略を変更できる。

### 5.2 イベントバスの設計要件

- イベントはイミュータブルな値オブジェクトとして定義する。1ファイルにまとめて一覧性を確保する
- ハンドラの登録・解除・発火ができるシンプルなインメモリ pub/sub で十分
- 同期ハンドラ（ドメイン状態更新用、即実行）と非同期ハンドラ（外部APIコール用、バックグラウンド実行）を区別する
- emit時に自動でログを記録する（9章参照）
- アプリ起動時に全ハンドラ登録の一覧をログ出力する機能を持つ。これがイベント駆動の「地図」になる

### 5.3 イベント登録のガイドライン

イベントの具体的な一覧は詳細設計で洗い出す。

- **同期ハンドラ**: ドメイン状態の更新（進捗記録、SRS更新など）およびUI通知（ViewModel経由）。1つのハンドラの失敗が他のハンドラを止めないようにする
- **非同期ハンドラ**: 外部APIコール（TTS生成、LLM呼び出しなど）。PySide6.QtAsyncio の統合イベントループ上で asyncio タスクとして実行
- **レジストリの一覧出力**: アプリ起動時に全ハンドラの登録状況をログに出す。これがイベント駆動の「地図」になる

## 6. LLMコール管理

### 6.1 Port/Adapter粒度の設計方針

**Portはドメインの要求単位で分割する。** ドメインは「欲しい結果」をPortに要求し、それを何回のAPIコールで実現するか、どの外部サービスを使うかはAdapter内部の判断である。この原則はLLMに限らず、TTS・音声評価など全てのPortに共通する。

```
ドメインが欲しいもの（Port）       Adapter内部の実装判断
─────────────────────          ──────────────────────
FeedbackResult                 ← 1回のLLMコールか2回に分けるか
PassageEvaluation              ← Azure Pronunciation Assessment + difflib
TTSAudio                       ← ElevenLabs API
```

ドメインは Port に対して「結果をください」とだけ要求する。Adapter 内部の実装詳細（APIコールの回数、モデル選択、分割戦略等）はドメインコードやテストに影響しない。

Port・Adapterの具体的な一覧と粒度は詳細設計で決定する。

### 6.2 LLMレスポンスのバリデーションとリトライ戦略

現在の主要LLM APIは**構造化出力（Structured Output）** を強制するモードを備えている。JSONスキーマを指定すればAPIレベルでそのスキーマに準拠したレスポンスが保証されるため、JSONパース失敗を考慮する必要はない。

ただし、以下の2点には対処が必要:

**1. 意味的バリデーション**: 構造は正しいが値がおかしいケース（SRSステージが負数、項目数が0件、空文字列など）。構造化出力で得られた結果に対して、ドメイン制約に基づく妥当性検証を行う。

**2. API通信エラーへのリトライ**: Rate Limit（429）、サーバーエラー（5xx）、タイムアウト等はいつでも発生しうる。各Adapterで指数バックオフによるリトライを設定する。共通基盤として抽象化する必要はない。Adapterの数は少なく（Gemini、ElevenLabs、Azure程度）、リトライ戦略もAdapter固有の調整が必要になるため、各Adapterに個別に設定するほうがシンプルで柔軟。

## 7. 外部サービス統合

### 7.1 サーバー側サービス（Port & Adapter で抽象化）

以下はサーバー側のサービスであり、ヘキサゴナルの Port/Adapter パターンで抽象化する。Provider 変更時に Adapter の差し替えで対応できる。

| サービス | 用途 | 参照 |
|---|---|---|
| Gemini（LLM） | パッセージ生成、フィードバック、判定 等 | [10-llm-calls.md](../requirements/10-llm-calls.md) |
| ElevenLabs（TTS） | 模範解答の音声生成 | [技術検証 V10](../verification/v10-tts/definition.md) |
| Azure Speech Service（発音評価） | 本番発話の合否判定、オーバーラッピング遅れ検知 | [技術検証 V9](../verification/v9-overlapping-detection/definition.md), [V11](../verification/v11-full-passage-evaluation/definition.md) |

これらに対応する Port・Adapter の具体的な分割は詳細設計で決定する（6.1節参照）。

### 7.2 音声デバイス（クライアント層に配置）

音声デバイス（マイク・スピーカー）はサーバー側サービスとは**根本的に性質が異なる**。

| | サーバー側サービス（Gemini等） | 音声デバイス |
|---|---|---|
| 呼び出し元 | サーバー側 | クライアント側 |
| モバイル移行後 | そのまま残る | 全面書き直し（AVFoundation等） |
| 抽象化の意味 | Provider差し替え | 差し替え先がない（プラットフォーム固有） |

サーバーは音声を録音しない。クライアントが録音し、`AudioData`（バイト列 + フォーマット + 長さ）としてドメインに渡す。したがって音声デバイスは Port/Adapter ではなく**クライアント層（UI）の一部**として配置する。

UIは ViewModel パターンを採用し、ドメインとUIの橋渡しを行う。ViewModel は Event Bus のハンドラとしてドメインイベントを受け取り、Qt Signal で UI に通知する（3.5節参照）。

**ルール算出とリアルタイム制御の分離**: ドメインはパラメータとルールを算出する（例: 「CEFR B1、85単語だからタイマーは12秒」）。その実行（12秒後に録音を止める）はクライアント側の責務。将来のモバイル移行時、ルール算出はサーバーに残り、リアルタイム制御はクライアントに留まる。

## 8. データストレージ戦略

### 8.1 DB選定: SQLite

プロトタイプ期は SQLite を使用する。

**選定理由:**
- シングルユーザー、デスクトップアプリ、ローカル完結。同時接続は1
- ファイル1つ。バックアップはコピー、リセットは削除
- 標準ライブラリで動く。追加の依存やインフラ不要
- コーディングエージェントが直接 SQL で参照できる
- 数万件のクエリはマイクロ秒で完了

将来のAPI移行時は、ヘキサゴナルの Repository Port を通じて Adapter を差し替える。

### 8.2 JSONカラム戦略

頻繁に変わるフィールドは JSON カラムに逃がし、マイグレーションの痛みを最小化する。**検索に使うフィールドは通常カラムとして定義し、それ以外（タグ、ヒント履歴、LLM抽出結果など）は JSON カラムに格納する。** プロトタイプ期のスキーマ変更の多くがマイグレーション不要になる。

### 8.3 「壊して作り直す」パターン

プロトタイプ期のデータ資産は蓄積された学習データではなく、LLMログとプロンプト履歴（DBの外に保存される）。テーブル構造を変更した際は DB を drop & recreate し、シードデータを投入して復旧する。シードデータはgit管理し、検証シナリオ（各SRSステージにばらけた項目、過去セッション履歴など）として育てていく。

段階的に堅くする方針:
- **プロトタイプ初期**: reset_db + seed（数秒で再構築）
- **プロトタイプ後期**: 簡易マイグレーション（バージョン番号 + SQLスクリプト）
- **本番移行時**: Alembic等の本格マイグレーションツール導入

### 8.4 音声ファイルの管理

**方針:**
- 命名規則は詳細設計で決定する（後から変えると過去データとの整合が壊れるため、実装前に確定させる）
- WAV（分析用）は分析完了後に削除、MP3（保存用）は永続保存
- ドメインモデルにはファイルパスではなく論理IDを持たせる。ファイル解決はインフラ層で行う

## 9. 可観測性（ロギング）

### 9.1 ログ基盤の方針

LLMの非決定的な出力とイベント駆動の因果関係の追いにくさに対処するため、3種類の構造化ログを整備する。ログ基盤ツール（Grafana Loki等）は導入しない。**JSONL ファイルをコーディングエージェントが直接 Read/Grep する**のが最もシンプルで効果的。

### 9.2 3種類のログ

すべて構造化ログ（JSONL: 1行1イベント）で出力する。grep/jqで即座にフィルタ・集計でき、コーディングエージェントもそのまま読める。

| 種類 | 記録内容 | 用途 |
|---|---|---|
| **イベントトレース** | 全ドメインイベントの発火・ハンドラ実行。セッションIDで紐付け | 因果チェーンの追跡。「何が何をトリガーしたか」 |
| **LLMコールログ** | コールID、モデル名、トークン数、レイテンシ、プロンプトハッシュ、バリデーション成否。リクエスト/レスポンス全文は別ファイルに保存 | プロンプト改善、コスト把握、レイテンシ分析 |
| **ドメイン状態変化** | エンティティの状態遷移（before/after + 理由） | 「なぜこの項目がステージ3にいるのか」の履歴追跡 |

ログの保存構造（ファイル命名規則等）は詳細設計で決定する。

### 9.3 ログ出力の自動挿入

ドメインコードにログ文をべた書きしない。EventBus のemit処理と、Port の呼び出し境界にミドルウェア/デコレータを挟み、通過する全イベント・全コールの入出力・レイテンシを自動記録する。ドメインコードにログ文を1行も書く必要がない。

### 9.4 開発支援CLIツール（任意）

セッションIDでのフィルタ、LLMコールの抽出、request/responseのペア表示ができる簡易CLIツールがあると便利。リッチなUIは不要。なくてもエージェントはJSONLを直接読める。

## 10. テスト戦略

### 10.1 3種類のテスト

**ドメインのユニットテスト（最重要）:** SRS間隔計算、セッション構成ルール、ヒントレベル決定など、ドメインロジックを外部依存なしでテストする。LLMもDBもモックなしで動く。これがヘキサゴナルの最大の実用的恩恵。

**LLMレスポンスのスナップショットテスト:** 実際のLLMレスポンスをJSONファイルとして保存し、バリデーション処理の入力に使う。LLMを叩かずにバリデーションロジックだけテストできる。新しいレスポンスパターンに遭遇したらfixtureに追加していく。

**LLMは叩かない壁:** CI やローカルの通常テストでは LLM を呼ばない。LLM を叩くのは手動の統合テストだけ。

### 10.2 プロトタイプ期にやらないこと

| やらない | 理由 |
|---|---|
| UIのテスト | pytest-qt で TDD。ViewModel、ウィジェット、ナビゲーション、View をテスト |
| E2Eテスト | セットアップコストが高い。ドメインテストでカバー |
| カバレッジ目標 | 数値を追うよりSRSとセッション進行の正しさに集中 |

## 11. 設定の外部化

### 11.1 外部化すべき設定項目

コード中にハードコードすると後で苦しむものを設定ファイルに集約する。

| カテゴリ | 項目例 | 理由 |
|---|---|---|
外部化の基準: **コードを変更せずに調整したい値**はすべて設定ファイルに出す。代表的なものとして SRSパラメータ、LLMモデル設定、タイマー設定、APIキー等があるが、これに限らない。詳細設計・実装の過程で必要に応じて追加していく。APIキーは `.env` に分離し、gitに含めない。

ドメインは設定値を起動時に受け取り、実行中は変更しない。設定の読み込みはインフラ層で行い、ドメインには値として渡す。
