## 1. はじめに

### 目的

本ドキュメントは AI-Driven-English アプリケーションの実装アーキテクチャを定義する。要件定義（[docs/requirements/](../requirements/README.md)）で定めた機能を、保守性・拡張性・テスト容易性を保ちながら実現するための構造的な指針を示す。

### 前提

- **プロトタイプフェーズ**: 機能の追加・削除・改善が頻繁に行われる。過度な抽象化は避け、実装速度を優先する
- **将来のモバイル移行**: デスクトップ（PySide6）→ API サーバー（FastAPI）+ モバイルクライアント（SwiftUI）への移行を見据える。ドメインロジックとAIオーケストレーションはサーバー側でそのまま再利用する
- **単一開発者 + コーディングエージェント**: 人間とAIエージェントの双方がコードを理解・修正できる構造にする

### このアプリの構造的な難しさ

このアプリケーションには3つの軸が交差する本質的な複雑さがある:

| 軸 | 内容 | 例 |
|---|---|---|
| ドメインロジック | ルールベースで決定論的な処理 | SRS間隔計算、セッション構成、定着判定、ヒントレベル制御 |
| AIオーケストレーション | LLMコールの管理・プロンプト制御 | 12種のLLMコール、事前生成/リアルタイム/イベント駆動の混在 |
| 外部サービス依存 | 複数の外部APIの統合 | Gemini（LLM）、ElevenLabs（TTS）、音声デバイス |

これらが密結合すると「SRSロジックを直したいだけなのにLLMコールのコードを触る必要がある」状態に陥る。本アーキテクチャの第一目的は、これら3軸を構造的に分離することである。

## 2. 設計原則

### 2.1 依存性逆転

ドメイン（ビジネスロジック）が最も重要であり、最も安定している。ドメインは外部の世界（UI、LLM、DB、音声デバイス）に一切依存しない。外部がドメインに依存する。

```
  ✗ Domain → Gemini API     （ドメインが外部を知っている）
  ✓ Domain → Port ← Adapter → Gemini API  （ドメインはPortだけ知っている）
```

### 2.2 3軸の分離

| 層 | 責務 | 変更理由 |
|---|---|---|
| Domain | ビジネスルール、状態管理 | 学習理論・アルゴリズムの改善 |
| AI Services / Orchestrator | プロンプト制御、LLMコールの分割・並列化 | プロンプト最適化、レイテンシ改善 |
| UI / Presentation | 画面描画、ユーザー操作の受付 | UX改善、プラットフォーム変更 |

これらを異なるディレクトリに配置し、依存方向を制限することで、各軸を独立に改善できる。

### 2.3 プロトタイプの実用主義

構造的に後から直すのが困難なものだけを今守り、それ以外はシンプルに保つ。

**今やること（後から直すのが困難）:**

- ドメインコアを外部依存から分離するディレクトリ構造
- LLMコールをPortの裏に隠す（ドメインにべた書きしない）
- ドメインイベントの仕組み（シンプルなインメモリ pub/sub で十分）
- UIとドメインの分離（UIフレームワークのコードにビジネスロジックを書かない）

**今やらないこと（プロトタイプが安定してから）:**

- リポジトリパターンによるDB抽象化の厳密な実装
- 厳密な集約境界の設計（まず動くモデルを作り、境界は後から調整）
- CQRS（コマンド/クエリ分離）の厳密な実装
- 非同期メッセージキュー（インメモリの非同期処理で十分）

## 3. 全体構造：ヘキサゴナルアーキテクチャ

### 3.1 概要

ヘキサゴナルアーキテクチャ（Ports & Adapters パターン）を採用する。ドメインを中心に配置し、外部世界との接点を Port（インターフェース）として定義する。Port の具体的な実装が Adapter である。

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  ┌──────────────────┐  ┌──────────┐  ┌───────────┐      │
│  │ Client           │  │ Gemini   │  │ ElevenLabs│      │
│  │ (UI + Audio)     │  │ Client   │  │ TTS Client│      │
│  └────┬─────────────┘  └────┬─────┘  └─────┬─────┘      │
│       │                     │              │             │
│       │ AudioData      ─────┼──────────────┼──           │
│       │ (値オブジェクト)      ▼              ▼             │
│       │               ┌──────────┐  ┌──────────┐        │
│       │               │ LLM Port │  │ TTS Port │        │
│       │               └────┬─────┘  └────┬─────┘        │
│       │                    │              │              │
│  ═════╪════════════════════╪══════════════╪═══           │
│       ▼                    ▼              ▼              │
│  ┌──────────────────────────────────────────────────┐    │
│  │              Domain Core（内側）                    │    │
│  │                                                  │    │
│  │  Knowledge: Source / Item / SRS                   │    │
│  │  Session:   Session / Block / Phase               │    │
│  │                                                  │    │
│  │  ※ドメインは AudioData を値として受け取るだけ        │    │
│  │    録音・再生の手段は知らない                        │    │
│  └──────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘
```

**音声デバイスとサーバー側サービスの違い:**
Gemini・ElevenLabs はサーバー側のサービスであり、Portで抽象化してAdapter差し替えが可能（Provider変更に対応）。一方、音声デバイス（マイク・スピーカー）はクライアント側のハードウェアであり、プラットフォームごとに全面書き直しになる。そのため、Portとして抽象化する意味がなく、クライアント層（UI）に含める。ドメインは録音結果を `AudioData`（バイト列 + フォーマット + 長さ）という値オブジェクトとして受け取るだけで、録音手段を知らない。
```

### 3.2 採用理由

| 恩恵 | このプロジェクトでの意味 |
|---|---|
| UI差し替え | PySide6 → FastAPI への移行時、ドメインとAIオーケストレーションをそのまま持っていける |
| LLMプロバイダ差し替え | Gemini → Claude 等の変更がAdapter層で完結する |
| テスト容易性 | ドメインロジックをLLMやDBなしで単体テストできる |
| プロンプト改善の独立性 | LLMのプロンプト変更がドメインロジックのテストを壊さない |

### 3.3 層の構成と依存方向

```
依存の方向: 外側 → 内側（内側は外側を知らない）

UI ──→ Domain   ✓（UIがドメインを呼ぶ）
Domain ──→ UI   ✗（ドメインはUIを知らない）
Adapter ──→ Port ✓（AdapterがPortを実装する）
Domain ──→ Port  ✓（ドメインはPortを通じて外部と話す）
Domain ──→ Adapter ✗（ドメインは具体的な実装を知らない）
```

### 3.4 ディレクトリ構成

```
src/parla/
├── knowledge/          ← Bounded Context A: 素材・学習項目・SRS
│   ├── domain/             エンティティ、値オブジェクト、ドメインサービス
│   └── services/           SRSスケジューリング、ストック判定 等
│
├── session/            ← Bounded Context B: セッション進行・フェーズ管理
│   ├── domain/             セッション、ブロック、フェーズのステートマシン
│   └── services/           メニュー構成、フィードバック統括、ブロック実行 等
│
├── reporting/          ← 読み取り専用の集計クエリ群（コンテキストではない）
│
├── ports/              ← Port: LLM、TTS、Repository等のインターフェース定義
├── adapters/           ← Adapter: Gemini、ElevenLabs、SQLite等のPort実装
├── orchestrator/       ← LLMコールの並列/直列制御
│
├── ui/                 ← クライアント層
│   ├── screens/            画面レイアウト
│   ├── audio/              録音・再生・波形（プラットフォーム固有）
│   └── viewmodels/         ドメインとUIの橋渡し
│
├── event_bus.py        ← イベントバス
└── config.py           ← 設定読み込み
```

## 4. ドメイン境界：境界づけられたコンテキスト

### 4.1 設計判断の背景

DDD（ドメイン駆動設計）の戦術的パターンを適用し、ドメインを2つの境界づけられたコンテキスト（Bounded Context）に分割する。

当初は4コンテキスト（Source Management / Learning Session / Mastery & SRS / Progress & History）を検討したが、以下の問題から2コンテキストに統合した:

- **Source と Mastery の双方向依存**: SRSバリエーション生成に別ソースの文脈が必要（Mastery→Source）など、当初の分離案では双方向依存が発生していた
- **Phase B フィードバックの3コンテキスト横断**: 入力（Passage + 音声 + 既存学習項目）と出力（動的模範解答 + 新規学習項目 + 練習記録）が3コンテキストに散らばり、変更のたびに3箇所を同時修正する必要があった
- **Progress & History の独自ドメインモデル不在**: 読み取り専用の集計処理であり、ビジネスルールを持たないため、コンテキストに昇格させると空のドメイン層が生まれる

### 4.2 Knowledge コンテキスト

**責務**: 「何を学び、どれだけ身についているか」の管理

#### エンティティと集約

**Source 集約:**

| エンティティ | 説明 |
|---|---|
| Source | ユーザーが登録したテキスト。ライフサイクル: 登録→生成中→未着手→学習中→完了→アーカイブ |
| Passage | Sourceから生成された学習単位（~100語） |
| Sentence | Passage内の個別発話単位（5-8文/passage） |
| PreAnswer | パッセージ生成時に作られるCEFR準拠の模範解答 |
| Hint | 文ごとのヒント（2段階）。生成後は不変 |

**LearningItem 集約:**

| エンティティ | 説明 |
|---|---|
| LearningItem | 具体的な表現パターン（例: "A is more X than B"）。タグ: 大分類/小分類/ドメイン |
| SRSRecord | ステージ(0-6)、間隔、信頼度乗数、次回レビュー日 |
| PracticeHistory | 練習記録（日時、正誤、WPM、タイマー消費率、ヒント使用段階） |

#### 主要なドメインサービス

- **SRSScheduler**: レビューキューの構築。ステージに基づく間隔計算、信頼度乗数の適用（[06-srs-mastery.md](../requirements/06-srs-mastery.md)）
- **ItemStocker**: 学習項目のストック判定。高信頼度の自動ストック、意味的重複検出（[05-learning-items.md](../requirements/05-learning-items.md)）
- **VariationGenerator**: SRSレビュー用のバリエーション文生成。別ソースからの文脈選択（[06-srs-mastery.md](../requirements/06-srs-mastery.md)）

### 4.3 Session コンテキスト

**責務**: 「今日の学習を駆動する」実行エンジン

#### エンティティと集約

**Session 集約:**

| エンティティ | 説明 |
|---|---|
| Session | 1日の学習セッション。3ブロックで構成 |
| SessionPlan | 前日にLLMが構成したメニュー（ブロック内容、推定時間） |
| Block | ブロック種別（Review / New / Consolidation）と進行状態 |
| PhaseState | パッセージごとのフェーズ進行（A→B→C）のステートマシン |
| DynamicAnswer | Phase Bでユーザーの発話から生成される模範解答。セッションスコープの成果物 |

#### 主要なドメインサービス

- **SessionComposer**: メニュー構成のオーケストレーション。SRSレビューキュー + ソース進捗を読み取り、LLMで翌日プランを生成（[03-session-structure.md](../requirements/03-session-structure.md)）
- **FeedbackOrchestrator**: Phase Bパイプラインの統括。音声→LLM→学習項目抽出→動的模範解答→ストック判定を調整（[02-learning-cycle.md](../requirements/02-learning-cycle.md)）
- **BlockRunner**: ブロック実行ロジック。ブロック間遷移、パッセージ進行、中断・再開管理（[03-session-structure.md](../requirements/03-session-structure.md)）

### 4.4 Reporting（インフラ関心事）

**位置づけ**: 境界づけられたコンテキストではない。独自のドメインモデルやビジネスルールを持たず、Knowledge と Session のデータを読み取り専用で集計するクエリ関数群。

**責務**:
- セッションサマリーの構成（所要時間、発話語数、新規ストック項目数、レビュー正答率）
- WPMトレンドの計算
- 学習カレンダーデータの集約
- カテゴリ別定着進捗の集計

将来、分析ロジックが複雑化した場合（例: 学習傾向から弱点を推定するアルゴリズム）、独立したコンテキストに昇格させる。

### 4.5 コンテキスト間の連携

コンテキスト間の依存は**1方向**に集約される:

```
Session ──────────────────→ Knowledge
  reads:  SRSレビューキュー、ソース進捗、学習項目一覧
  writes: 新規学習項目、練習記録、ヒント使用イベント
```

連携手段:
- **読み取り**: Session が Knowledge のサービスを Port 経由で呼び出す
- **書き込み**: Session がドメインイベント（例: `ItemExtracted`）を発火し、Knowledge 側のハンドラが受け取ってストック処理を行う

## 5. イベント駆動アーキテクチャ

### 5.1 ドメインイベントの役割

このアプリはバックグラウンドでの先読み生成（[11-latency-strategy.md](../requirements/11-latency-strategy.md)）が生命線である。「何をトリガーに何を事前生成するか」はプロトタイプ中に頻繁に変わるため、トリガーと後続処理を疎結合にする必要がある。

イベント駆動にしない場合、Phase B 完了処理はこうなる:

```python
# 悪い例: Phase Bが後続処理を全て知っている
def complete_phase_b(self, passage_id):
    feedback = self.generate_feedback(passage_id)
    self.tts_service.generate(feedback.dynamic_answer)
    self.coherence_service.regenerate(feedback)
    self.variation_service.generate(feedback.new_items)
    self.progress_service.update(feedback)
```

イベント駆動にすると:

```python
# Phase Bは「終わった」と言うだけ
def complete_phase_b(self, passage_id):
    feedback = self.generate_feedback(passage_id)
    self.emit(PhaseBCompleted(passage_id, feedback))
```

後続処理はハンドラとして独立に登録・削除できる。Phase B のコードに触れずにバックグラウンド生成戦略を変更できる。

### 5.2 イベントバスの設計要件

- イベントはイミュータブルな値オブジェクトとして定義する。全コンテキストから参照されるため、1ファイルにまとめて一覧性を確保する
- ハンドラの登録・解除・発火ができるシンプルなインメモリ pub/sub で十分
- 同期ハンドラ（ドメイン状態更新用、即実行）と非同期ハンドラ（外部APIコール用、バックグラウンド実行）を区別する
- emit時に自動でログを記録する（9章参照）
- アプリ起動時に全ハンドラ登録の一覧をログ出力する機能を持つ。これがイベント駆動の「地図」になる

### 5.3 主要なイベントとハンドラの対応

| イベント | トリガー | ハンドラ（同期） | ハンドラ（非同期） |
|---|---|---|---|
| `SourceRegistered` | ソース登録時 | — | パッセージ生成（LLM #1） |
| `SentenceRecorded` | Phase A 文発話完了 | 次文タイマー開始 | フィードバック生成（LLM #4） |
| `PhaseBCompleted` | Phase B 完了 | 進捗更新 | TTS生成 |
| `PassageCompleted` | パッセージ完了 | パッセージ状態更新 | Block 3用バリエーション生成（LLM #3） |
| `ItemExtracted` | Phase Bでの項目抽出 | Knowledge側でストック処理 | — |
| `MenuConfirmed` | 翌日メニュー確定 | — | パッセージ生成、バリエーション一括生成、素材再構成 |
| `SessionEnded` | セッション終了 | セッション履歴保存 | 翌日メニュー構成（ルールベース）、復習類題生成 |

### 5.4 イベント登録のガイドライン

- **同期ハンドラ**: ドメイン状態の更新（進捗記録、SRS更新など）。1つのハンドラの失敗が他のハンドラを止めないようにする
- **非同期ハンドラ**: 外部APIコール（TTS生成、LLM呼び出しなど）。バックグラウンドで実行
- **レジストリの一覧出力**: アプリ起動時に全ハンドラの登録状況をログに出す。これがイベント駆動の「地図」になる

## 6. LLMオーケストレーション

### 6.1 Port粒度の設計方針

**Port の粒度 ≠ LLMコールの粒度**。ドメインは「欲しい結果」をPortに要求し、それを何回のLLMコールで実現するかはAdapter内部の判断である。

```
ドメインが欲しいもの（Port）       実際のLLMコール（Adapter内部）
─────────────────────          ──────────────────────
FeedbackResult                 ← 1回で取れるかもしれないし
  - reproduction                  2回に分けるかもしれない
  - dynamic_answer
  - extracted_items            ← 出力が不安定なら別コールにする
```

ドメインは Port に対して「FeedbackResult をください」とだけ要求する。それを1回のLLMコールで取るか、出力安定性のために2回に分割するかは Adapter 内部の判断。この設計により、LLMコールの分割・統合がAdapter内部で閉じ、ドメインコードやテストに影響しない。

### 6.2 Orchestrator層

LLMコール間の依存関係に基づく並列/直列制御を担う。Adapter層の内部に位置する。

```
ドメイン: PhaseBCompleted イベント発火
    │
    ▼
Orchestrator:
    │
    ├─ step 1: coherence_regen(sentences)  ← まず統合模範解答を生成
    │           ↓ 結果が出たら
    ├─ step 2: 並列実行
    │     ├─ tts_generate(coherent_answer)  ← TTS生成
    │     └─ variation_gen(items)           ← バリエーション生成
    │           ↓ 全部揃ったら
    └─ 結果をドメインに返す / イベント発火
```

依存グラフ（DAG）としてコール間の関係を表現する。「AとBは並列、Cは両方の完了後」という構造を宣言的に記述し、非同期ランタイム（asyncio等）で実行する。

**変更の影響範囲:**

| やりたい変更 | 触る場所 |
|---|---|
| LLMコールを分割 | Adapter内部だけ |
| 並列化戦略を変更 | Orchestratorだけ |
| Gemini → Claude に変更 | Adapterだけ |
| ドメインロジック修正 | Domain だけ |

### 6.3 LLMレスポンスのバリデーションとリトライ戦略

現在の主要LLM APIは**構造化出力（Structured Output）** を強制するモードを備えている。JSONスキーマを指定すればAPIレベルでそのスキーマに準拠したレスポンスが保証されるため、JSONパース失敗を考慮する必要はない。

ただし、以下の2点には対処が必要:

**1. 意味的バリデーション**: 構造は正しいが値がおかしいケース（SRSステージが負数、項目数が0件、空文字列など）。構造化出力で得られた結果に対して、ドメイン制約に基づく妥当性検証を行う。

**2. API通信エラーへのリトライ**: Rate Limit（429）、サーバーエラー（5xx）、タイムアウト等はいつでも発生しうる。各Adapterで指数バックオフによるリトライを設定する。共通基盤として抽象化する必要はない。Adapterの数は少なく（Gemini、ElevenLabs程度）、リトライ戦略もAdapter固有の調整が必要になるため、各Adapterに個別に設定するほうがシンプルで柔軟。

## 7. 外部サービス統合

### 7.1 サーバー側サービス（Port & Adapter で抽象化）

以下はサーバー側のサービスであり、ヘキサゴナルの Port/Adapter パターンで抽象化する。Provider 変更時に Adapter の差し替えで対応できる。

| サービス | 用途 | Port | 参照 |
|---|---|---|---|
| Gemini（LLM） | パッセージ生成、フィードバック、判定 等 | `LLMPort` | [10-llm-calls.md](../requirements/10-llm-calls.md) |
| ElevenLabs（TTS） | 模範解答の音声生成 | `TTSPort` | [技術検証 V10](../verification/v10-tts/definition.md) |

### 7.2 音声デバイス（クライアント層に配置）

音声デバイス（マイク・スピーカー）はサーバー側サービスとは**根本的に性質が異なる**。

| | サーバー側サービス（Gemini等） | 音声デバイス |
|---|---|---|
| 呼び出し元 | サーバー側 | クライアント側 |
| モバイル移行後 | そのまま残る | 全面書き直し（AVFoundation等） |
| 抽象化の意味 | Provider差し替え | 差し替え先がない（プラットフォーム固有） |

サーバーは音声を録音しない。クライアントが録音し、`AudioData`（バイト列 + フォーマット + 長さ）としてドメインに渡す。したがって音声デバイスは Port/Adapter ではなく**クライアント層（UI）の一部**として配置する。

```
ui/
├── screens/           ← 画面レイアウト
├── audio/             ← 音声デバイス制御（プラットフォーム固有）
│   ├── recorder.py       録音 → AudioData 返却
│   ├── player.py         再生・速度変更
│   └── waveform.py       波形・レベルメーター描画
└── viewmodels/        ← ドメインとUIの橋渡し
```

**各層の責務:**

| 処理 | 層 | 理由 |
|---|---|---|
| マイク録音の開始・停止 | クライアント（ui/audio） | プラットフォーム固有のデバイスI/O |
| 音声再生・速度変更 | クライアント（ui/audio） | プラットフォーム固有のデバイスI/O |
| 波形・レベルメーター描画 | クライアント（ui/audio） | リアルタイム表示 |
| WAV→MP3変換 | インフラ（infra） | ファイル処理。サーバー側でも使える |
| 「録音データをLLMに送る」判断 | Session ドメイン | ビジネスロジック |
| 「タイマーは12秒」の算出 | Session ドメイン | ビジネスルール（CEFR + WPM目標から計算） |
| 「12秒経ったので録音を止める」 | クライアント | リアルタイム制御 |

**ルール算出とリアルタイム制御の分離**: ドメインはパラメータとルールを算出する（例: 「CEFR B1、85単語だからタイマーは12秒」）。その実行（12秒後に録音を止める）はクライアント側の責務。将来のモバイル移行時、ルール算出はサーバーに残り、リアルタイム制御はクライアントに留まる。

## 8. データストレージ戦略

### 8.1 DB選定: SQLite

プロトタイプ期は SQLite を使用する。

**選定理由:**
- シングルユーザー、デスクトップアプリ、ローカル完結。同時接続は1
- ファイル1つ。バックアップはコピー、リセットは削除
- 標準ライブラリで動く。追加の依存やインフラ不要
- コーディングエージェントが直接 SQL で参照できる
- 数万件のクエリはマイクロ秒で完了

将来のAPI移行時は、ヘキサゴナルの Repository Port を通じて Adapter を差し替える。

### 8.2 JSONカラム戦略

頻繁に変わるフィールドは JSON カラムに逃がし、マイグレーションの痛みを最小化する。**検索に使うフィールドは通常カラムとして定義し、それ以外（タグ、ヒント履歴、LLM抽出結果など）は JSON カラムに格納する。** プロトタイプ期のスキーマ変更の多くがマイグレーション不要になる。

### 8.3 「壊して作り直す」パターン

プロトタイプ期のデータ資産は蓄積された学習データではなく、LLMログとプロンプト履歴（DBの外に保存される）。テーブル構造を変更した際は DB を drop & recreate し、シードデータを投入して復旧する。シードデータはgit管理し、検証シナリオ（各SRSステージにばらけた項目、過去セッション履歴など）として育てていく。

段階的に堅くする方針:
- **プロトタイプ初期**: reset_db + seed（数秒で再構築）
- **プロトタイプ後期**: 簡易マイグレーション（バージョン番号 + SQLスクリプト）
- **本番移行時**: Alembic等の本格マイグレーションツール導入

### 8.4 音声ファイルの管理

**命名規則**（後から変えると過去データとの整合が壊れるため、最初に決める）:

```
audio/
├── recordings/
│   └── {session_id}/{passage_id}/sentence_{n}_attempt_{m}.wav
├── tts/
│   └── {passage_id}/dynamic_answer.mp3
└── converted/
    └── {session_id}/{passage_id}/sentence_{n}_attempt_{m}.mp3
```

**ライフサイクル:**
- WAV（分析用）: LLMへの送信・分析完了後に削除
- MP3（保存用）: 変換後に永続保存。学習項目詳細画面での再生に使用

**ドメインモデルにはファイルパスではなく論理IDを持たせる。** ファイル解決はインフラ層（AudioStore）で行う。

## 9. 可観測性（ロギング）

### 9.1 ログ基盤の方針

LLMの非決定的な出力とイベント駆動の因果関係の追いにくさに対処するため、3種類の構造化ログを整備する。ログ基盤ツール（Grafana Loki等）は導入しない。**JSONL ファイルをコーディングエージェントが直接 Read/Grep する**のが最もシンプルで効果的。

### 9.2 3種類のログ

すべて構造化ログ（JSONL: 1行1イベント）で出力する。grep/jqで即座にフィルタ・集計でき、コーディングエージェントもそのまま読める。

| 種類 | 記録内容 | 用途 |
|---|---|---|
| **イベントトレース** | 全ドメインイベントの発火・ハンドラ実行。セッションIDで紐付け | 因果チェーンの追跡。「何が何をトリガーしたか」 |
| **LLMコールログ** | コールID、モデル名、トークン数、レイテンシ、プロンプトハッシュ、バリデーション成否。リクエスト/レスポンス全文は別ファイルに保存 | プロンプト改善、コスト把握、レイテンシ分析 |
| **ドメイン状態変化** | エンティティの状態遷移（before/after + 理由） | 「なぜこの項目がステージ3にいるのか」の履歴追跡 |

ログの保存構造: 1セッション1ファイル（`logs/sessions/{date}_{session_id}.jsonl`）。LLMコールのリクエスト/レスポンス全文は `logs/llm_calls/` に個別保存。

### 9.3 ログ出力の自動挿入

ドメインコードにログ文をべた書きしない。EventBus のemit処理と、Port の呼び出し境界にミドルウェア/デコレータを挟み、通過する全イベント・全コールの入出力・レイテンシを自動記録する。ドメインコードにログ文を1行も書く必要がない。

### 9.4 開発支援CLIツール（任意）

セッションIDでのフィルタ、LLMコールの抽出、request/responseのペア表示ができる簡易CLIツールがあると便利。リッチなUIは不要。なくてもエージェントはJSONLを直接読める。

## 10. テスト戦略

### 10.1 3種類のテスト

**ドメインのユニットテスト（最重要）:** SRS間隔計算、セッション構成ルール、ヒントレベル決定など、ドメインロジックを外部依存なしでテストする。LLMもDBもモックなしで動く。これがヘキサゴナルの最大の実用的恩恵。

**LLMレスポンスのスナップショットテスト:** 実際のLLMレスポンスをJSONファイルとして保存し、バリデーション処理の入力に使う。LLMを叩かずにバリデーションロジックだけテストできる。新しいレスポンスパターンに遭遇したらfixtureに追加していく。

**LLMは叩かない壁:** CI やローカルの通常テストでは LLM を呼ばない。LLM を叩くのは手動の統合テストだけ。

### 10.2 プロトタイプ期にやらないこと

| やらない | 理由 |
|---|---|
| UIのテスト | UIは頻繁に変わる。手動確認で十分 |
| E2Eテスト | セットアップコストが高い。ドメインテストでカバー |
| カバレッジ目標 | 数値を追うよりSRSとセッション進行の正しさに集中 |

## 11. 設定の外部化

### 11.1 外部化すべき設定項目

コード中にハードコードすると後で苦しむものを設定ファイルに集約する。

| カテゴリ | 項目例 | 理由 |
|---|---|---|
| SRSパラメータ | ステージ別間隔、信頼度乗数（ヒント段階別） | チューニング対象。頻繁に調整する |
| LLMモデル設定 | モデルID、temperature、max_tokens、用途別モデル使い分け | モデル切り替えを気軽にやりたい |
| タイマー設定 | CEFR別目標WPM、バッファ係数 | 学習理論の調整に連動 |
| APIキー | Gemini、ElevenLabs等 | .envに分離。gitに含めない |

ドメインは設定値を起動時に受け取り、実行中は変更しない。設定の読み込みはインフラ層で行い、ドメインには値として渡す。
