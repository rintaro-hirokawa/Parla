# V11: 本番発話の内容評価 — 検証結果 (2026-04-08)

## 1. 背景と目的

### Phase C 本番発話とは

Phase C では、学習者はオーバーラッピング練習の直後に、日本語のお題のみを見て英語でパッセージ全体を発話する。模範解答に近い発話が期待される。

### V11 の問い

> Phase C の本番発話（~60秒）を、実用的な速度で合否判定し、差分情報を返せるか？

### 合格基準

- 合否判定精度: 85% 以上
- レイテンシ: 発話終了後 5 秒以内に結果表示
- 差分情報が UI 表示に十分な品質

---

## 2. 検証の経緯

### Trial 1: LLM ベース（2段階パイプライン）

**アーキテクチャ:**

```
音声 → [Stage 1] Gemini Flash Lite（書き起こし）→ [Stage 2] Gemini Flash（添削評価）→ 結果
```

**結果:**

| 指標 | 結果 | 目標 | 判定 |
|------|------|------|------|
| 合否判定精度 | 100% (4/4) | >= 85% | ✅ |
| 合計レイテンシ中央値 | **17.01s** | <= 5s | ❌ |
| Stage 1 中央値 | 4.76s | — | — |
| Stage 2 中央値 | 12.24s | — | — |

**問題点:**

1. **レイテンシ大幅超過**: Stage 2（テキストのみの LLM 評価）が 12s。8文の構造化出力 + reasoning_effort=medium が原因
2. **Stage 1 の書き起こし不安定**: 同一音声で 3 回実行しても書き起こし結果が異なる。特に fail_partial ケースで、スキップした文を LLM が補完（ハルシネーション）する問題が発生
3. **fail_partial の判定不安定**: 3回中 2回で PASS と誤判定（中央値は FAIL で最終結果は正解だが、一貫性に欠ける）

**教訓:**

- LLM に書き起こしをやらせるべきではない。LLM は「補完」が本能で、無音を埋めてしまう
- テキストのみの LLM 評価でも、構造化出力が重いと 10s+ かかる
- V5（リトライ判定）の教訓「各ツールに得意なことだけやらせる」が改めて確認された

### 方針転換: Azure Speech Service ストリーミング

Trial 1 の結果を受け、以下の設計変更を行った:

1. **LLM による書き起こしを廃止** → Azure Speech Service の Pronunciation Assessment に変更
2. **ストリーミング送信** → 発話中にリアルタイムで音声を送り、発話終了後の待ち時間を最小化
3. **LLM による添削評価を廃止** → difflib によるプログラム判定に変更
4. **日本語混入テストを廃止** → Phase C では日本語発話を UI レベルで不可とする設計決定

## 3. 最終的な技術構成

### パイプライン

```
ユーザー発話（マイク）
    │
    ├── PushAudioInputStream でリアルタイムにチャンク送信（100ms 単位）
    │
    ↓
Azure Pronunciation Assessment（ストリーミング、連続認識モード）
    │   reference_text = 模範解答全文
    │   enable_miscue = False（30秒超のため）
    │
    ├── recognized コールバックで utterance ごとに結果蓄積
    │   - 認識テキスト
    │   - 単語ごと AccuracyScore / ErrorType
    │
    ↓ 発話終了（stream.close()）
    ↓ 最終 recognized コールバック到着（~1s）
    │
difflib.SequenceMatcher による後処理（~0.5ms）
    │   - Omission / Insertion 検出（V9 と同一ロジック）
    │   - 文ごとの similarity 計算
    │   - status 判定: correct (>= 0.90) / paraphrase (0.50-0.90) / error (< 0.50 or Omission > 50%)
    │
    ↓
結果表示  発話終了から合計 ~1.2s
```

### ストリーミングの仕組み

WAV ファイルを 100ms チャンク（3200 bytes @ 16kHz/16bit/mono）に分割し、リアルタイム速度で `PushAudioInputStream` に送信する。これはマイク入力と同じ API・同じタイミングであり、プロダクションのユースケースを正確に再現している。

```python
while offset < len(pcm_data):
    chunk = pcm_data[offset:offset + CHUNK_SIZE]
    stream.write(chunk)
    offset += CHUNK_SIZE
    # リアルタイム速度で送信
    elapsed = time.perf_counter() - stream_start
    expected = offset / bytes_per_second
    if expected > elapsed:
        time.sleep(expected - elapsed)
```

Azure は音声を受け取りながら並行処理し、utterance の切れ目（ポーズ）ごとに `recognized` コールバックを発火する。そのため、発話終了時点ではほとんどの処理が完了しており、最後の utterance の処理待ち（~1秒）だけが体感レイテンシとなる。

### V9 との関係

V9（オーバーラッピング遅れ検知）では Azure Pronunciation Assessment をファイルベースで使用し、34秒の音声に 17秒かかっていた。本検証でストリーミング方式に切り替えたことで、V9 のレイテンシ問題の解決策も同時に示された。

## 5. 検証結果

### 合否判定精度: 100% (4/4)

| ケース | 期待 | 実際 | ストリームレイテンシ | Completeness | 判定 |
|--------|------|------|---------------------|--------------|------|
| pass_fluent | PASS | PASS | 1.45s | 98.0% | ✅ |
| pass_paraphrase | PASS | PASS | 0.52s | 97.5% | ✅ |
| fail_omission | FAIL | FAIL | 0.16s | 100.0% | ✅ |
| fail_semantic | FAIL | FAIL | 0.47s | 91.3% | ✅ |

3回実行全てで同一の合否判定。LLM ベース（Trial 1）で見られた判定の揺れは発生しなかった。

### レイテンシ

| 指標 | 結果 | 目標 |
|------|------|------|
| ストリームレイテンシ中央値 | **0.49s** | <= 2.0s ✅ |
| ストリームレイテンシ最大値 | **1.45s** | — |
| 後処理（difflib + 判定） | **0.0005s** | — |
| **合計体感レイテンシ** | **~1.2s** | <= 5.0s ✅ |

Trial 1（LLM ベース、17s）と比較して **約 14 倍の高速化**。

### 文ごとの判定品質

**pass_fluent**: 全8文 similarity=1.00。模範通りの発話を完璧に認識。

**pass_paraphrase**: 全8文 PASS 判定。ただし言い換え箇所は Azure が Omission 扱いにする:
- 文4: "really hard" → 認識されず、similarity=0.92（correct 判定は正しい）
- 文7: "little by little" → "by" のみ認識、similarity=0.93
- 文8: "Because of" → "Because" 脱落、similarity=0.93

**fail_omission**: 文6 を正しく error 判定。similarity=0.00、"(no speech)" として検出。

**fail_semantic**: 文8 を正しく error 判定。"Toyota is a very big company in Japan." と話したが、Azure は reference_text と大幅に異なるため大部分を Omission 扱い。similarity=0.24。

## 6. 制約と設計上の割り切り

### Azure Pronunciation Assessment の特性

Azure は `reference_text` と一致する単語を優先的に認識する。そのため:

- ✅ 模範通りに話した → 完璧に認識（similarity=1.00）
- ✅ 文をスキップした → Omission として正しく検出
- △ 言い換えた → 実際のテキストではなく Omission 扱い（ただし similarity で合格判定は可能）
- △ 全く違う内容を話した → ゴミテキスト（ただし低 similarity で不合格判定は可能）

### 割り切りの根拠

Phase C はオーバーラッピング練習の直後であり、学習者は模範解答に近い発話をすることが期待される。「模範通りに言えたか否か」の二値判定が本質的な要件であり、言い換え箇所の正確な復元は必須ではない。

## 7. 将来的な改善案: Whisper 併用による添削 UI の実現

現状の Azure 単体では「ユーザーが実際に何と言ったか」のテキストが正確に取れない。添削 UI（ユーザーの発話を表示し、差分箇所に赤線を引く）を実現するには、以下の拡張が有効:

```
ユーザー発話（マイク）
    │
    ├── [並列 Track A] Azure ストリーミング → 合否判定 + Omission 検出
    │
    └── [並列 Track B] Whisper ストリーミング → 正確な書き起こし（自由認識）
    │
    ↓ 発話終了
    │
    ├── Azure 結果 + Whisper 書き起こしを統合
    │
    └── LLM（Gemini Flash, minimal）で添削分析:
        - Whisper テキスト × 模範解答 → diff_segments 生成
        - Azure の AccuracyScore → 発音品質情報
        - 「ここを言い換えましたね」等の参考情報生成
    │
    ↓
添削 UI 表示（ユーザーの発話テキスト + 赤線 + 模範との差分）
```

この拡張により:
- Azure の強み（Omission 検出、発音評価、低レイテンシ）を活用しつつ
- Whisper の強み（正確な自由認識、多言語対応）で書き起こし品質を補完
- LLM は最小限の判定（paraphrase vs error の分類）のみ担当

両方ともストリーミングで並列実行すれば、追加レイテンシは LLM 判定の ~1-2s のみ。合計 ~3s で添削 UI 向けの結果を返せる見込み。

## 8. 結論

### 合格基準の達成状況

| 基準 | 結果 | 判定 |
|------|------|------|
| 合否判定精度 >= 85% | 100% (4/4) | ✅ PASS |
| 発話終了後 5秒以内 | ~1.2s | ✅ PASS |
| 差分情報が UI 表示に十分か | 合否判定 + Omission 検出は十分。言い換え箇所の正確な復元は不可 | △ 条件付き PASS |

### 総合判定: PASSED（条件付き）

Phase C の本番発話評価として、「模範通りに言えたか否か」の合否判定は Azure ストリーミング + プログラム判定で実用レベルを達成。添削 UI（ユーザー発話の可視化 + 差分表示）は将来的に Whisper 併用で対応可能。

### 技術選定の確定

| 用途 | 技術 | 根拠 |
|------|------|------|
| 音声認識 + 評価 | Azure Pronunciation Assessment（ストリーミング） | 低レイテンシ（~1s）、Omission 検出、3回実行で安定 |
| 差分判定 | difflib.SequenceMatcher | 0.5ms で完了、LLM 不要 |
| 書き起こし強化（将来） | Whisper ストリーミング（並列） | 自由認識で正確なテキスト復元 |
| 添削分析（将来） | Gemini Flash（minimal） | Whisper + Azure の結果統合 |
