# V5: リトライ判定の速度と精度 — 検証レポート (2026-04-07)

## 1. 背景と目的

### フェーズBの即時リトライとは

Parla のフェーズBでは、学習者が日本語のみ表示された状態で、直前に見た動的模範解答を英語で再発話する。この再発話をシステムが「正解/不正解 + 一言コメント」として即時判定し、不正解の場合は最大3回リトライさせる。

このフィードバックループが遅いとテンポが崩れ、学習体験が大幅に劣化する。LLM コール #5 はこのリアルタイム判定を担う。

### V5 の問い

> フェーズBの即時リトライで、1センテンスの発話音声を「正解/不正解 + 一言コメント」として3秒以内に判定でき、かつその判定は妥当か?

### 合格基準（definition.md より）

| 基準 | 目標値 |
|------|--------|
| 応答時間（中央値） | ≤ 3秒 |
| 応答時間（最大値） | ≤ 5秒 |
| 正誤判定の正答率 | ≥ 85% |

---

## 2. 検証環境

- **LLM 呼び出し:** LiteLLM 経由、Gemini API 直接（Proxy なし）
- **構造化出力:** Pydantic モデルを `response_format` に指定
- **音声送信:** base64 エンコードした WAV を data URI で Gemini に直接入力（STT 経由しない）
- **レイテンシ計測:** `time.perf_counter()` で `litellm.completion()` のみを計測
- **各テストケースを3回実行**し、中央値のレイテンシと判定結果を代表値として使用

### テストデータ

5シナリオ × 4音声タイプ = 20ケース。各シナリオはB1レベルの文法パターンを対象とし、開発者自身が録音した実音声を使用した。

| シナリオ | 学習項目 | 動的模範解答 |
|---------|---------|-------------|
| S01 | A is more X than B | This problem is more difficult than yesterday's problem. |
| S02 | have/has + past participle | I have been to Italy three times. |
| S03 | If + present, will + verb | If it rains tomorrow, I will stay at home. |
| S04 | be + past participle (passive) | This book has been translated into many languages. |
| S05 | used to + verb | I used to play in the park every day when I was a child. |

### 音声タイプと期待判定

| タイプ | 内容 | 期待判定 |
|--------|------|---------|
| correct | 模範解答とほぼ同一の発話 | 正解 |
| alt_correct | 別の表現だが意味は同じ | 不正解（模範解答の再現が目的のため） |
| partial | 前半は正しいが途中で日本語混入・崩壊 | 不正解 |
| incorrect | 学習項目の構文を使っていない | 不正解 |

## 3. 検証の経緯

本検証は4段階の試行を経て、最終的なモデル・プロンプト構成にたどり着いた。各段階で何を試し、何がうまくいかず、何を学んだかを記録する。

### 試行1: 複雑な評価プロンプト + reasoning_effort=high（デフォルト）

**構成:**
- モデル: `gemini/gemini-3-flash-preview`
- reasoning_effort: 指定なし（デフォルト = high）
- プロンプト: 学習項目、日本語お題、模範解答、CEFRレベルを全て入力し、3つの評価基準（学習項目使用→最重要、意味カバー、英語として意味が通じるか）を優先順位付きで指示
- 出力: `{correct, reason (英語20語), item_used}`
- alt_correct の期待判定: 正解（学習項目を使えていれば表現違いも許容）

**結果:**

```
正答率:           19/20 (95.0%)
alt_correct正答率: 5/5 (100.0%)
レイテンシ中央値:  3.84s  ← 目標 ≤3.0s を超過
レイテンシ最大値:  10.66s ← 目標 ≤5.0s を大幅超過
```

**分析:**
- 精度は良好だが、レイテンシが大幅に未達
- partial ケースで 10秒超のスパイクが集中（日本語混在音声の処理に時間がかかっている可能性）
- Gemini 3 Flash はデフォルトで `thinking_level=high` で動作しており、このような単純な判定タスクに対して不必要な推論トークンを消費している

**学び:** thinking level の最適化が必要。

### 試行2: reasoning_effort=low

**変更点:** `reasoning_effort="low"` を指定（Gemini 3 Flash では `thinking_level=low` にマッピングされる）

**結果:**

```
正答率:           16/20 (80.0%)  ← 目標 ≥85% を下回った
alt_correct正答率: 5/5 (100.0%)
レイテンシ中央値:  2.53s  ← 目標達成
レイテンシ最大値:  3.03s  ← 目標達成
```

**分析:**
- レイテンシは劇的に改善（中央値 3.84s → 2.53s、スパイクも解消）
- しかし正答率が80%に低下。partial ケースの4/5で「学習項目は使えているから正解」と判定してしまった
- 推論を減らしたことで「文が崩壊している」という判断ができなくなった

**学び:** reasoning_effort だけでは精度とレイテンシを両立できない。プロンプトの根本的な見直しが必要。

### 試行3（最終）: プロンプトの抜本的簡素化 + reasoning_effort=minimal

**設計思想の転換:**

フェーズBの即時リトライは、学習者が**直前に見た動的模範解答を再現できるかの確認**である。複雑な「学習項目の使用判定」や「意味的正解の許容」は不要で、**模範解答との一致度だけ見ればよい**。

この気づきにより、以下を変更した:

1. **プロンプトの大幅簡素化:**
   - 旧: 学習項目、日本語お題、模範解答、CEFRレベル、3段階の評価基準を詳細に説明
   - 新: 「この音声がターゲット文と一致しているか判定せよ。冠詞・三単現の-s・短縮形は許容。それ以外の違いは不一致。」（4行）

2. **入力トークンの削減:**
   - 旧: 学習項目説明 + 日本語お題 + 模範解答 + 評価基準
   - 新: 模範解答テキスト1文のみ

3. **出力の簡素化:**
   - 旧: `{correct, reason (英語20語), item_used}`
   - 新: `{correct, reason (日本語15文字)}`
   - `item_used` を廃止（模範解答と一致すれば学習項目は自動的に含まれるため不要）
   - reason を日本語に変更（UIに表示する可能性を考慮）

4. **alt_correct の期待判定を変更:**
   - 旧: 正解（学習項目を使えていれば許容）
   - 新: 不正解（模範解答と異なる単語を使っている → リトライさせる）
   - 理由: フェーズBは記憶からの再現練習であり、創造的な言い換えを評価するフェーズではない

5. **reasoning_effort を minimal に:**
   - Gemini 3 Flash の `minimal` は "no thinking に最も近い" 設定
   - 完全無効化はできないが、ほぼ思考しない

**最終プロンプト:**

```
音声がターゲット文と一致しているか判定せよ。
許容: 冠詞(a/an/the)の欠落・誤用、三単現-sの欠落、短縮形(I'll=I will)。
不一致: 上記以外の単語の違い、語の欠落、日本語の混入、文の未完成。
出力: JSON {"correct": bool, "reason": "日本語15文字以内"}
```

```
ターゲット文: {reference_answer}
```

**結果:**

```
正答率:           19/20 (95.0%)  ← 目標達成
レイテンシ中央値:  2.37s          ← 目標達成
レイテンシ最大値:  3.14s          ← 目標達成
総合判定: PASS
```

**詳細結果:**

| シナリオ | タイプ | 期待 | 判定 | レイテンシ | reason |
|---------|-------|------|------|-----------|--------|
| S01 | correct | 正解 | 正解 | 2.68s | 一致しています |
| S01 | alt_correct | 不正解 | 不正解 | 2.47s | 複数の語彙が異なる |

### 試行4（参考）: gemini-3.1-flash-lite-preview

より軽量なモデルで同等以上の結果が得られるか検証した。

**構成:**
- モデル: `gemini/gemini-3.1-flash-lite-preview`
- reasoning_effort: high
- プロンプト: 試行3と同じ（簡素版）

**結果:**

```
正答率:           17/20 (85.0%)
レイテンシ中央値:  3.07s  ← 目標超過
レイテンシ最大値:  8.10s  ← 目標大幅超過
```

**分析:**
- S01_correct（正解音声）を「単語が異なります」と誤判定する致命的エラーが発生
- S03, S05 の alt_correct を正解と判定する不安定さ
- レイテンシもむしろ遅く（flash より遅い）、2秒〜9秒とばらつきが大きい
- flash-lite はプレビュー版のため不安定な可能性があるが、このタスクには力不足と判断

**学び:** flash-lite は精度・レイテンシともに flash に劣り、このタスクには不適。

## 5. 発見事項と知見

### プロンプト設計

1. **タスクの本質を正確に捉えた単純なプロンプトが最強:** フェーズBの即時リトライは「模範解答の再現確認」であり、「学習項目の使用判定」や「意味的正解の許容」は過剰な複雑さだった。タスクの本質を正確に捉え、判定ルールを明確に限定したことで、精度とレイテンシの両方が改善した。

2. **許容範囲は具体的に列挙する:** 「minor grammar errors」のような曖昧な指示ではなく、「冠詞、三単現の-s、短縮形」と具体的に列挙することで、判定のブレが大幅に減少した。

3. **不一致条件を明示する:** 「日本語の混入」「文の未完成」を不一致条件として明示することで、partial ケースの誤判定が解消された。

### Gemini の thinking 設定

4. **Gemini 3 Flash はデフォルトで thinking_level=high:** 単純なタスクでもデフォルトでは不必要に推論トークンを消費し、レイテンシが増大する。レイテンシが重要なタスクでは `reasoning_effort` の明示的な設定が必須。

5. **thinking を下げるだけでは精度が低下する:** 複雑なプロンプトのまま thinking を下げると、微妙な判断（partial ケースの正誤判定）ができなくなる。プロンプトの簡素化と thinking の削減は**セットで行う**必要がある。

6. **Gemini 3 では thinking を完全に無効化できない:** `minimal` が最小設定であり、"no thinking に最も近い" が完全OFFではない。Gemini 2.5 Flash では `thinking_budget=0` で完全無効化可能。

### モデル選定

7. **flash-lite は音声判定タスクに力不足:** 正解音声を不正解と誤判定する致命的エラーが発生し、レイテンシも flash より遅かった。音声を含む multimodal タスクでは flash 以上のモデルが必要。

### 判定基準の設計

8. **フェーズBの判定は「模範解答との一致」に割り切る:** 「意味的に正しければ表現が違ってもOK」という柔軟な判定は、フェーズBでは不要。学習者は直前に模範解答を見ており、再現練習のフェーズだからである。表現の多様性を評価するのは、フェーズCやブロック復習の役割。

### 唯一の誤判定

9. **S03 alt_correct: "I'll stay home" を正解と判定:** 模範解答 "I will stay at home" に対し、`I'll = I will` は許容ルールに含めているため短縮形自体は正しい判定。ただし `at home → home` の省略も同時に発生しており、これを許容するかは判断が分かれる。実用上は問題にならないレベル。

## 7. 残存リスクと今後の課題

1. **プレビューモデルでの検証:** `gemini-3-flash-preview` での結果であり、GA版ではレイテンシ・精度が変動する可能性がある。GA版リリース時に再検証が望ましい。

2. **テストデータの多様性:** 5シナリオ×4タイプ=20ケースでの検証であり、より多様な文法パターン・文長・発話スタイルでの追加検証が望ましい。

3. **ネットワークレイテンシ:** 開発環境（日本）からのAPI呼び出しで計測。本番環境のネットワーク構成によりレイテンシが変動する。

4. **許容範囲の微調整:** `at home → home` のような省略をどこまで許容するか、実運用でのユーザーフィードバックを踏まえてチューニングが必要。
