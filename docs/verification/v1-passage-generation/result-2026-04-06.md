# V1: パッセージ生成品質 — 検証レポート

## 1. 背景と目的

### Parla におけるパッセージ生成の位置づけ

Parla は日本語話者向けの AI 駆動英語スピーキング学習アプリである。学習フローは「ソーステキスト登録 → パッセージ生成 → 発話練習 → フィードバック → SRS」という流れで進行する。

パッセージ生成（LLM コール #1）はこのフローの入口にあたり、ここで生成される学習素材の品質が、以降のフィードバック・SRS・通し練習すべてのフェーズに連鎖的に影響する。そのため、LLM にパッセージ生成を任せるにあたって、実用に耐える品質が安定的に得られるかを事前に検証する必要がある。

### 不確実性

本検証で明らかにしたい不確実性は以下の通り（`docs/verification/v1-passage-generation/definition.md` より）:

- ソースの種類（特に口語的で構造の緩い YouTube 要約）によって出力品質がどれだけ変動するか
- LLM が「最適なパッセージ数・分量を決定する」設計で、実用的な分割ができるか
- CEFR レベルの制御が効くか（B1 指定で本当に B1 相当の難易度になるか）
- ヒント（2段階）が段階的な情報開示として機能するか
- パッセージ形式（説明型）の指定が意図通り反映されるか

### 検証パターン

最もリスクが高い組み合わせで検証した:

| 項目 | 値 |
|------|-----|
| ソース | YouTube 動画の要約テキスト（日本語、約3000字） |
| CEFR レベル | B1 |
| 英語バリエーション | American English |
| パッセージタイプ | 説明型 |
| LLM モデル | `gemini/gemini-3.1-pro-preview` |

---

## 2. 検証環境

- LLM 呼び出し: [LiteLLM](https://docs.litellm.ai/) 経由（Proxy なし、Langfuse なし）
- 構造化出力: Pydantic モデルを `response_format` に指定し、JSON スキーマで出力を強制
- バリデーション失敗時: 最大2回リトライ

### 出力スキーマ

```
PassageGenerationResult
├── source_summary: str
└── passages: list[Passage]
    ├── passage_index: int
    ├── topic: str
    ├── passage_type: str
    └── sentences: list[Sentence]
        ├── ja: str         # 日本語お題
        ├── en: str         # 英語模範文
        └── hints: Hint
            ├── hint1: str  # 先頭語 + キーワード
            └── hint2: str  # 構文スケルトン（日本語文法用語）
```

### 改善の詳細

#### 試行 1→2: hint2 の日本語化

- **課題**: hint2 が全て英語の文法用語（"Subject + be-verb + complement"）で、日本語話者にとって馴染みが薄い
- **対策**: 日本語文法用語（主語、be動詞、動詞(現在形)、that節 等）を使用するようプロンプトに指示
- **結果**: `主語 + be動詞 + 補語` のように安定して日本語化された

#### 試行 2→3: 節の親子構造の可視化

- **課題**: that節や関係代名詞節の内部構造が平坦で、どこからどこまでが節なのか判別しにくい
- **対策**: 節の内部構造を `[]` で括り、文法ラベル `()` と区別するルールを導入
- **結果**: `主語 + 動詞(現在形) + that節[主語 + 助動詞 + 動詞(原形) + 目的語]` のように入れ子が明確になった

#### 試行 3→4: 文法用語の正確性

- **課題**: 「接続詞節」という不正確な用語の使用、until を接続詞として扱う誤りが発生
- **対策**: 副詞節は具体的な接続詞名で表記（when節、if節、because節 等）。until/before は前置詞として正しく扱うよう明記
- **結果**: `when節[主語 + 動詞(現在形)]` のように正確な用語で出力されるようになった

#### 試行 4→5: 文章量の増加

- **課題**: 3パッセージ・21文・約280語では学習素材として量が不足
- **対策**: 1パッセージあたり約120-180語・7-10文、3000字のソースに対して5-6パッセージを目安に指定
- **結果**: 5パッセージ・40文・635語に増加。品質の劣化は見られず、1回の LLM 呼び出しで対応可能

#### 試行 5→6: 語彙レベル制御と日英対応

- **課題**: (1) B1 レベルに対して "chauffeur" のような C1 相当の語彙が出現。(2) 日本語と英語で主語が対応しないケースが1箇所発生
- **対策**: 「迷ったら第二言語話者に馴染みのある語彙を選ぶ」制約と「日英で主述の対応を保つ」制約をプロンプトに追加
- **結果**: "chauffeur" → "driver" に改善。主語の不一致も解消

## 5. 発見事項と注意点

### プロンプト設計に関する知見

1. **文法用語の正確性は明示的に指定する必要がある**: LLM は「接続詞節」のような不正確な用語や、until を接続詞として扱うなどの誤りを犯す。正しい品詞分類を具体例とともにプロンプトに書く必要がある。

2. **語彙レベルの制御には具体的なガイダンスが有効**: 「B1 レベルを使え」だけでは不十分で、「迷ったら第二言語話者に馴染みのある語彙を選べ」という方針を明示することで、chauffeur → driver のような改善が得られた。

3. **日英対応の制約は明示しないとズレる**: 特に受動態・使役・倒置のような構文変換が絡むと、日英で主語がずれやすい。制約として明記することで安定した。

4. **文章量を倍増させても品質は維持された**: 3パッセージ→6パッセージへの増加で、1回の LLM 呼び出しのまま品質を維持できた。呼び出し分割は不要。

### 残存リスク

- **パッセージタイプ**: 今回は「説明型」のみ検証。「対話型」「意見型」では異なる課題が出る可能性がある
- **ソースの多様性**: YouTube 要約（日本語）1件のみで検証。英語記事、技術文書、短文など他のソース種別では未検証
- **モデル依存**: `gemini/gemini-3.1-pro-preview` での検証結果であり、他モデルでは出力品質が異なる可能性がある
