# レイテンシ対策

本ドキュメントでは、LLMコールやTTS生成に伴うレイテンシを最小化するための戦略を定義する。

---

## 基本原則

1. **先読み生成**: ユーザーが必要とする前にバックグラウンドで生成を開始する
2. **パイプライン処理**: ユーザーが現在の作業中に次の準備を並行して進行する
3. **段階的表示**: 全部揃う前に準備できたものから順に表示する
4. **クライアント側即時判定**: LLM不要な判定はローカルで即時実行する

---

## 処理フローごとの分析

### ソース登録 → パッセージ生成

- 想定所要時間: 30秒〜2分
- 方式: 非同期処理。登録完了後にバックグラウンドで生成
- ユーザー影響: 低。生成完了まで当該ソースは「生成中」ステータスとなるが、学習開始は別ソースで可能

### セッション構成

- 想定所要時間: 即時（決定論的ルールベースのため）
- 方式: メニュー確定時にルールに基づき構成。LLMコール不要
- ユーザー影響: なし

### ブロック1 類題生成

- 想定所要時間: 1問あたり3〜8秒、20問で60〜160秒
- 方式: 前日にバッチ生成済み
- ユーザー影響: なし

### フェーズA → B（フィードバック生成）

- 想定所要時間: 1センテンスあたり5〜15秒
- 方式: センテンスごとにバックグラウンド並列実行。ユーザーが次のセンテンスを発話している間に、先行センテンスのフィードバック生成を完了させる
- ユーザー影響: 先行センテンスのフィードバックは待ちなし。最後のセンテンスのフィードバックのみ待ちが発生しうるが、先に完了済みのフィードバックを表示開始することで体感待ち時間を短縮する

### フェーズB → C（TTS生成）

- 想定所要時間: 100ワードで3〜10秒
- 方式: フィードバック表示開始と同時にTTS生成を開始する
- ユーザー影響: ユーザーがフィードバック確認およびストック操作を行っている間に完了する

### フェーズC オーバーラッピング（遅れ検知）

- 想定所要時間: [技術検証 V9](../verification/v9-overlapping-detection/definition.md)
- 方式: オーバーラッピング実施直後にバックグラウンド処理
- ユーザー影響: 技術検証 V9 の結果次第

### フェーズC 本番発話（内容評価）

- 想定所要時間: ~1.2秒（発話終了後）
- 方式: 時間内完了の判定はクライアント側で即時実行（WPMとタイマーベース）。内容評価は Azure Speech Service のストリーミング Pronunciation Assessment で発話中にリアルタイム処理し、発話終了後 ~1.2秒で結果を返す（[技術検証 V11](../verification/v11-full-passage-evaluation/definition.md)）
- ユーザー影響: 時間判定は即時。内容評価もほぼ即時（~1.2秒）

### ブロック2 → 3（定着類題生成）

- 想定所要時間: 1問あたり3〜8秒
- 方式: 各パッセージ完了時に即座にバックグラウンド生成を開始する
- ユーザー影響: パッセージ学習中に順次生成が完了するため、ブロック3開始時には大半が準備済み

## バックグラウンド処理のタイミング一覧

| タイミング | 処理内容 |
|-----------|---------|
| 前日セッション完了時 | セッション構成（ルールベース）、復習類題(#3)、素材再構成(#11) |
| ソース登録時 | パッセージ生成(#1) |
| フェーズA中 | フィードバック生成(#4) ※センテンスごと並列 |
| フェーズB中 | TTS生成 |
| パッセージ完了時 | 定着類題生成(#3) |
| タスクトレイ常駐 | 毎日深夜にメニュー鮮度チェックおよび再生成 |
